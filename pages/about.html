In my own words

I started in L&D the way most people do — building slides, recording voiceovers, packaging SCORM files. But I kept running into the same ceiling: the tools stopped where the problems didn't.

So I learned to write JavaScript inside Storyline. Then to connect those courses to live APIs. Then to automate the workflows that used to eat entire project weeks. Not because I wanted to be a developer — but because the learner experience I wanted to create required it.

Today I sit in a specific niche: I think like an instructional designer and build like an engineer. I understand pedagogy and performance objectives, and I can also write a Cloudflare Worker proxy, script FFmpeg across 8 languages, or wire an xAPI stream to an LRS. That combination is rare, and it's where I do my best work.

Journey

EARLY CAREER
Getting the fundamentals right

Started in L&D with classic Instructional Design work — storyboarding, scripting, building in early-era authoring tools. Learned quickly that the gap between "what I want to build" and "what the tools let me build" was the most interesting problem in the field.

MID CAREER
Moving into Storyline & JavaScript

Began extending Articulate Storyline with JavaScript — custom triggers, dynamic content, live variable manipulation. What started as workarounds became a systematic approach: write the interaction logic in JS, use Storyline as a rendering layer. This changed what was possible.

Approach

How I actually
think about this work.

Most L&D professionals start with content. I start with behaviour — what the learner needs to do differently after, and what's currently stopping them from doing it. Everything else follows from that.

I build things I'd want to use myself. If I'm bored building it, the learner will be bored using it. If I'm proud of it, they usually notice.

  • →
    Technology is a means, not a signal I use complex tech when it solves a real problem, and simple tech when that's what the problem needs. xAPI isn't impressive for its own sake — it's impressive when the data changes what you do next.
  • →
    Build once, run everywhere I default to reusable systems over one-off solutions. Every template, component, and automation I build is documented well enough that someone else can maintain and extend it.
  • →
    The learner is always the user I test everything from the learner's perspective first. Stakeholder approval and SME sign-off matter, but they're not the measure of success. Behaviour change is.
  • →
    If you can automate it, you probably should Manual repetition is a signal that there's a systems problem waiting to be solved. I'd rather spend two days automating a five-hour weekly task than keep doing the five-hour task.
